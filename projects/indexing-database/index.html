Nền tảng về Index

Index là gì?

Hãy tưởng tượng bạn có một cuốn từ điển dày 5.000 trang.

  • Không có Index (Full Scan):

    Bạn phải lật từng trang từ trang 1 đến trang 5.000 để tìm từ "Zebra". Rất lâu!

  • Có Index (B-Tree Lookup):

    Bạn lật mục lục, tìm chữ "Z", thấy trang 4950. Lật thẳng đến đó. Rất nhanh.

3

Index hỗ trợ gì ngoài WHERE?

Lọc (Filtering)

Các phép toán so sánh chính xác (=) hoặc phạm vi (>, <, BETWEEN).

WHERE age > 18

Sắp xếp (Sorting)

Dữ liệu trong Index đã được sắp xếp sẵn. DB không cần tốn CPU để sort lại.

ORDER BY created_at DESC

Kết nối (Joining)

Tăng tốc độ ghép đôi các dòng giữa hai bảng thông qua khóa ngoại.

JOIN orders ON u.id = o.user_id

Tại sao DB "bơ" Index của tôi?

Thủ phạm: Query Optimizer

Nó là "kế toán trưởng". Trước khi chạy, nó tính chi phí. Nếu dùng Index mà tốn công hơn (phải nhảy qua nhảy lại quá nhiều), nó sẽ chọn quét toàn bộ bảng (Full Scan).

Vấn đề độ chọn lọc (Selectivity) Nếu bạn tìm 'Giới tính = Nam' (50% dữ liệu), Index vô dụng. Chỉ Index cột có giá trị đa dạng (Email, ID).
Dùng hàm lên cột Index WHERE YEAR(dob) = 1990 ❌ -> Index bị vô hiệu hóa vì dữ liệu bị biến đổi.

Sai lầm & Tuyệt kỹ (Covering Index)

Sai lầm thường gặp

  • Đánh Index "cho chắc" Thấy cột nào trong WHERE cũng đánh. Hậu quả: Tốn ổ cứng, chậm thao tác INSERT/UPDATE.
  • Index trên cột quá dài Ví dụ cột 'Mô tả' (Text). Index sẽ rất cồng kềnh.
  • Tạo nhiều Index đơn lẻ Thay vì 1 Composite Index (A, B), bạn tạo 1 Index A và 1 Index B. Hiệu quả kém hơn.

Tuyệt kỹ: Covering Index

Khái niệm: Index chứa TẤT CẢ dữ liệu mà câu SQL cần. Database chỉ cần đọc Index rồi trả kết quả luôn, không cần sờ vào bảng dữ liệu gốc.

SQL Query
SELECT name, phone FROM User WHERE email = 'a@b.com'
Giải pháp:
CREATE INDEX idx_covering ON User (email, name, phone);

-> Tốc độ nhanh nhất có thể!
