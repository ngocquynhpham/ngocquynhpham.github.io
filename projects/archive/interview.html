  • DX vượt trội: Dev server khởi động <1s bất kể project size, tăng năng suất team GDC toàn cầu.
TYPESCRIPT = BẮT BUỘC (TẠI SAO?)
  • Type-safe Data Schema: AI Platform xử lý data phức tạp (model outputs, predictions) — TS đảm bảo contract giữa Frontend và Backend API không bị lệch.
  • Refactor an toàn: Hệ thống lớn với nhiều dev (VN + Korea), TS giúp bắt lỗi compile-time thay vì runtime.

REACT ROUTER — ĐIỀU HƯỚNG AI PLATFORM
  • Nested Routes: AI Platform nhiều module (Dashboard, Models, Data, Settings) — React Router v6+ quản lý layout lồng nhau rất tốt.
  • Protected Routes: Phân quyền truy cập theo role (Admin, Analyst, Viewer) cho từng section.
  • Lazy Loading: Code-splitting theo route, chỉ load module user cần — tối ưu performance.
"UI PUBLISHER" LÀ GÌ? (THUẬT NGỮ HÀN/NHẬT)

UI Publisher: Thuật ngữ phổ biến trong công ty Hàn/Nhật, chỉ người chuyên chuyển đổi Design (Figma/PSD) thành HTML/CSS pixel-perfect. Tập trung vào markup chính xác và cross-browser compatibility.

Frontend Integrator: Bước tiến hóa từ Publisher — không chỉ cắt UI mà còn tích hợp logic, gọi API, quản lý state và đảm bảo data flow hoạt động đúng giữa Frontend & Backend.

Vai trò tại CJ: Kết hợp cả hai — vừa đảm bảo UI pixel-perfect theo Figma, vừa integrate component vào hệ thống React/TS phức tạp của AI Platform. Là cầu nối Design ↔ Engineering.

DESIGN-TO-CODE WORKFLOW (FIGMA → REACT)
  • Figma Inspect: Đọc specs trực tiếp (spacing, color tokens, typography) từ Figma Dev Mode để code chính xác.
  • Component Mapping: Map Figma components → Ant Design components, customize qua ConfigProvider & Design Tokens thay vì override CSS.
  • Responsive Desktop: JD focus Desktop — xử lý breakpoints cho các kích thước màn hình 1280px, 1440px, 1920px+ (không phải mobile-first).
  • Handoff Review: QA pixel-perfect bằng overlay tools, đảm bảo 1:1 với design trước khi merge.
RESTFUL API INTEGRATION
  • Axios/Fetch + TS: Định nghĩa Response Types chặt chẽ cho mỗi endpoint — tránh runtime error khi backend thay đổi schema.
  • API Layer Pattern: Tách riêng service layer (api/*.ts) khỏi UI logic, dễ mock khi dev và dễ swap endpoint.
  • Error Handling: Xử lý HTTP status codes (401, 403, 500) tập trung qua interceptors, hiển thị thông báo user-friendly.
  • Loading States: Skeleton/Spinner cho async data, optimistic updates cho form submissions.
COMPLEX FORMS & VALIDATION (KEY COMPONENT)
  • Ant Design Form: Hệ thống Form mạnh nhất React ecosystem — field validation, dynamic fields, form list, dependencies giữa các field.
  • Multi-step Forms: AI Platform thường có wizard flows (tạo model, config pipeline) — cần quản lý state across steps.
  • Validation Flows: Async validation (check trùng tên model từ API), conditional rules, custom validators cho data AI-specific.
  • Dynamic Fields: Thêm/xóa fields runtime (VD: config hyperparameters cho AI model — số lượng params thay đổi theo model type).
STATE MANAGEMENT — QUẢN LÝ DỮ LIỆU PHỨC TẠP
  • React Context + useReducer: Phù hợp cho shared state nhỏ (theme, auth, locale) — lightweight, không cần thêm library.
  • Zustand / Redux Toolkit: Khi state phức tạp (filter states, cached API data, multi-tab sync) — cần predictable state container.
  • Server State (React Query/SWR): Tách biệt server state khỏi client state — auto caching, revalidation, optimistic updates cho data-heavy dashboard.
  • Tại sao quan trọng: AI Platform có nhiều data source đồng thời (models, datasets, metrics) — state management tốt = UI không bị desync.
AI PLATFORM CỦA CJ — PHÂN TÍCH CONTEXT

AI Platform có thể bao gồm:

  • AI Dashboard: Monitor model performance, accuracy metrics, inference logs — cần Charts (ECharts/Recharts) + Data Tables nặng.
  • Data Annotation Tool: Giao diện gán nhãn dữ liệu cho training AI — complex forms + drag-and-drop UI.
  • Model Management: CRUD models, versioning, A/B testing config — nhiều dynamic forms và nested data.
  • Analytics Console: Phân tích kết quả AI predictions cho business teams (CGV, Logistics) — data visualization nặng.

Frontend challenges đặc thù:

  • Real-time data: WebSocket/SSE cho streaming AI inference results lên Dashboard.
  • Large datasets: Virtual scrolling, pagination server-side cho bảng hàng chục nghìn records.
  • Complex filters: Multi-dimensional filtering (theo model, date range, accuracy threshold, dataset) — cần UI/UX thông minh.
  • Role-based UI: Admin thấy full, Data Scientist thấy models, Business User thấy reports — cần component-level access control.

Framework Phân Tích Cinematic UI (5 Bước Chi Tiết)

01. Bố Cục (Layout & Grid)

Các loại phổ biến:

  • Z-Pattern: Di chuyển mắt theo chữ Z (phù hợp Landing page ít nội dung).
  • F-Pattern: Quét từ trái sang, trên xuống (Chuẩn cho Desktop Dashboard).
  • Rule of Thirds: Chia 3x3 để đặt điểm nhấn vào các giao điểm vàng.
  • Central/Symmetry: Bố cục trung tâm/đối xứng tạo sự vững chãi.
Mapping UI: Cách sắp xếp Sidebar, Topbar và Content Area giúp user Desktop vận hành mượt mà nhất.
02. Màu Sắc (Color Theory)

Lý thuyết áp dụng:

  • Semantic Colors: Đỏ (Error), Vàng (Warning), Xanh (Success).
  • Contrast (Tương phản): Độ đối lập sáng tối để dẫn dắt sự chú ý.
  • Color Palette: Phối màu đơn sắc (Monochromatic) hay bổ túc (Complementary).
Mapping UI: Dùng High-contrast cho các thông báo quan trọng của AI Platform để Power User không bỏ lỡ.
03. Ánh Sáng & Chiều Sâu (Depth)

Kỹ thuật phân tích:

  • Shadows/Elevation: Tạo khối cho vật thể, tách biệt với nền.
  • Layers: Các lớp không gian chồng lên nhau (Tiền cảnh/Hậu cảnh).
  • Focus Points: Vùng sáng nhất là vùng user cần thao tác.
Mapping UI: Dùng Z-index thông minh để quản lý Modals, Overlays và Context Menus.
04. Typography (Chữ & Phân Cấp)

Các yếu tố cần phân tích:

  • Hierarchy: Phân cấp rõ ràng giữa H1 (Tiêu đề phim), H2 (Tagline), Body (Credits, ngày chiếu).
  • Readability: Kích thước, khoảng cách dòng, trọng lượng font có đủ dễ đọc từ khoảng cách xa?
  • Personality: Font chữ có truyền tải đúng "tính cách" tác phẩm? (Serif = cổ điển, Sans = hiện đại, Display = cá tính).
  • Contrast Ratio: Chữ trên nền phức tạp có đủ tương phản để đọc được (WCAG AA)?
Mapping UI: Xây dựng Type Scale hệ thống (Design Tokens) để mọi Dashboard trong GDC dùng chung bộ font size, line-height và font-weight nhất quán.
05. Ẩn Dụ & Storytelling (Visual Metaphor)

Cách đọc ẩn dụ thị giác:

  • User Journey: Poster dẫn dắt mắt theo một "câu chuyện" — từ hook (điểm bắt mắt) đến climax (thông điệp chính) đến CTA (ngày chiếu).
  • Visual Metaphor: Hình ảnh biểu tượng gợi sự liên tưởng sâu hơn lời nói (VD: vệt che mắt = sự phân tầng giai cấp).
  • Emotional Trigger: Yếu tố nào kích hoạt cảm xúc? Sợ hãi, tò mò, hồi hộp hay phấn khích?
  • Easter Eggs: Chi tiết ẩn giấu khiến người xem quay lại nhìn kỹ hơn (tăng engagement).
Mapping UI: Áp dụng Storytelling vào Onboarding Flow và Empty States — biến trải nghiệm khô khan thành hành trình có cảm xúc cho user.

Mock Case Scenario-Based Interview

Case: “Smart Logistics Dashboard — Ngày Đầu Nhận Việc”

Tình huống giả lập: Bạn vừa join team GDC, được assign vào dự án Smart Logistics. Bạn chưa biết gì về hệ thống nội bộ CJ.

Bối cảnh Interviewer đưa ra:

“Tưởng tượng bạn vừa vào CJ GDC được 1 tuần. Team lead Korea giao cho bạn task đầu tiên: Build một trang Dashboard hiển thị realtime tracking của 50,000 kiện hàng/ngày cho hệ thống CJ Logistics. Dữ liệu từ API backend team Korea cung cấp. Design đã có trên Figma. Bạn sẽ approach task này như thế nào — biết rằng bạn chưa biết gì về codebase, conventions, hay internal tools của CJ?”

Q1: Approach — Bước đầu tiên bạn làm gì?

“Bạn chưa biết gì về hệ thống. Tuần đầu tiên bạn sẽ làm gì trước khi code?”

Gợi ý trả lời (thật thà, không nổ): “Em sẽ không code ngay. Tuần đầu em tập trung 3 việc:

1. Đọc hiểu codebase: Xem project structure, folder conventions, xem team đang dùng Ant Design version nào, có custom theme không, state management là gì (Redux? Zustand? Context?).

2. Đọc Figma kỹ: Xác định component nào đã có sẵn trong Design System team, component nào phải build mới. Em sẽ hỏi Designer rõ: bảng tracking này có bao nhiêu trạng thái kiện hàng? Có filter theo khu vực không? Có real-time hay polling?

3. Hỏi team: API docs ở đâu? Response format ra sao? Có mock data không? Convention đặt tên file, commit message, PR review process là gì?

Tại TechX em cũng từng join dự án giữa chừng — bài học lớn nhất là: hỏi sớm 1 ngày thì tiết kiệm được 1 tuần debug.”
Q2: Technical — 50,000 records, làm sao không lag?

“Dashboard hiển thị 50K kiện hàng. Bạn xử lý performance thế nào?”

Gợi ý trả lời: “Ở Thang Loi em đã xử lý bảng 10K+ records, nên em có kinh nghiệm thực tế:

1. Không render hết: Dùng Virtual Scroll — chỉ render ~50 rows visible trên viewport, DOM nhẹ hơn 200 lần.

2. Pagination phía server: 50K records thì Frontend không nên hold hết. Em sẽ confirm với Backend: API có hỗ trợ offset/limit hoặc cursor-based pagination không?

3. Memoization: React.memo cho row components, useMemo cho filter/sort logic, tránh re-render toàn bộ table khi 1 cell thay đổi.

4. Debounce search/filter: User gõ tìm kiếm → debounce 300ms trước khi call API, tránh spam requests.

Tuy nhiên, em chưa biết Ant Design Pro Table handle virtual scroll khác gì so với thư viện em từng dùng. Em sẽ cần thời gian spike 1-2 ngày để benchmark.”
Q3: Integration — Làm việc với team Korea

“API từ Backend Korea trả về format khác hoàn toàn với mock data Designer đưa. Bạn xử lý sao?”

Gợi ý trả lời: “Đây là tình huống em gặp thường xuyên ở cả TechX lẫn Thang Loi. Cách em xử lý:

1. Tạo Adapter Layer: Viết một lớp transform/mapper nằm giữa API response và UI components. Frontend luôn nhận data đúng format mà UI cần, bất kể Backend thay đổi gì.

2. Type-safe contract: Dùng TypeScript interface định nghĩa rõ APIResponse vs UIModel. Nếu Backend thay field name, TypeScript sẽ báo lỗi ngay lúc build — không cần chờ đến runtime.

3. Giao tiếp: Gửi tin nhắn rõ ràng cho Backend team: ‘Anh/chị ơi, em thấy field pkg_status trong API nhưng Figma ghi là delivery_state — confirm giúp em field nào là source of truth?’

Với em, Integrator không chỉ là code — là cầu nối giữa mọi người.”

WCAG Là Gì? Accessibility Trong Frontend

WCAG (Web Content Accessibility Guidelines)

Bộ tiêu chuẩn quốc tế giúp nội dung web có thể truy cập được bởi mọi người, bao gồm người khuyết tật. Phiên bản hiện tại: WCAG 2.1 / 2.2.

4 Nguyên tắc P.O.U.R
  • Perceivable: Nội dung phải nhận biết được (alt text, captions, contrast)
  • Operable: Giao diện phải thao tác được (keyboard navigation, no time traps)
  • Understandable: Nội dung phải hiểu được (clear language, predictable UI)
  • Robust: Tương thích với nhiều công cụ (assistive tech, screen readers)
Các mức tuân thủ
  • Level A: Mức tối thiểu (alt text, keyboard accessible)
  • Level AA: Mức khuyến nghị — đây là chuẩn CJ nên follow (contrast 4.5:1, focus visible)
  • Level AAA: Mức cao nhất (contrast 7:1, sign language cho video)
Ví dụ câu hỏi & cách trả lời: Q: “WCAG là gì? Bạn áp dụng thế nào?”
“WCAG là bộ tiêu chuẩn accessibility quốc tế. Em áp dụng Level AA: đảm bảo contrast ratio tối thiểu 4.5:1, dùng semantic HTML, thêm aria-label cho interactive elements, và test keyboard navigation cho toàn bộ flow.”

CSS Mixin, Variables & Preprocessor

Mixin là gì?

Một “function” trong SCSS/LESS cho phép tái sử dụng một nhóm declarations. Truyền tham số để tạo biến thể.

Variables

CSS Custom Properties (--color-primary) hoặc SCSS ($primary). Quản lý Design Tokens tập trung.

Extend vs Mixin

@extend gộp selectors (DRY). @mixin copy declarations (linh hoạt hơn, dùng với params).

Ví dụ giải thích code SCSS Mixin 
// Mixin: responsive font-size
@mixin responsive-text($min, $max, $min-vw: 320px, $max-vw: 1440px) {
  font-size: $min;
  @media (min-width: $min-vw) {
    font-size: calc(#{$min} + #{strip-unit($max - $min)}
      * (100vw - #{$min-vw}) / #{strip-unit($max-vw - $min-vw)});
  }
  @media (min-width: $max-vw) { font-size: $max; }
}
// Sử dụng:
h1 { @include responsive-text(24px, 48px); }
p  { @include responsive-text(14px, 18px); }

Giải thích: Mixin tạo fluid typography — font-size tự scale mượt theo viewport. Pattern phổ biến trong Design System.

Quốc Tế Hóa (i18n) Trong CSS & HTML

CJ là tập đoàn Hàn Quốc hoạt động toàn cầu. Hiểu i18n là điểm cộng lớn khi phỏng vấn.

  • i18n (Internationalization): Thiết kế code/UI sẵn sàng cho đa ngôn ngữ
  • L10n (Localization): Áp dụng ngôn ngữ/văn hóa cụ thể (dịch, format ngày/tiền, RTL)

Kỹ thuật i18n trong CSS & HTML

HTML:

  • • lang="ko" / lang="vi" trên thẻ html
  • • dir="rtl" cho Arabic/Hebrew
  • • <meta charset="UTF-8"> hỗ trợ ký tự đặc biệt
  • • Không hardcode text — dùng translation keys

CSS:

  • • :lang(ko) selector cho font-family riêng tiếng Hàn
  • • Logical properties: margin-inline-start thay margin-left
  • • text-overflow: ellipsis — text dài hơn khi dịch
  • • Flexible width — không fix width cứng cho text

MISC Quick Reference — Câu hỏi ngắn, đáp nhanh

Flashcard: Các Câu Hỏi Kỹ Thuật Có Thể Gặp

Q: Box Model trong CSS là gì?

Mỗi element = Content + Padding + Border + Margin. box-sizing: border-box để width tính luôn padding + border.

Q: Flexbox vs Grid — khi nào dùng gì?

Flexbox = 1 chiều (hàng HOẶC cột). Grid = 2 chiều (hàng VÀ cột). CJ Dashboard → Grid cho layout, Flex cho component.

Q: BEM là gì?

Block__Element--Modifier. Naming convention cho CSS. VD: .card__title--highlighted. Với CSS Modules / AntD, BEM ít cần hơn.

Q: Semantic HTML là gì?

Dùng đúng tag: <nav>, <main>, <article>, <aside>. Quan trọng cho SEO, Accessibility, và maintainability.

Q: CSS Specificity?

Inline (1000) > ID (100) > Class (10) > Element (1). !important ghi đè nhưng tránh dùng. AntD dùng Design Tokens thay vì override.

Q: Responsive vs Adaptive?

Responsive = fluid (%, vw). Adaptive = layout cố định cho breakpoints. CJ Desktop → Adaptive cho 1280/1440/1920px.

Q: Design Tokens là gì?

Giá trị thiết kế được đặt tên tập trung: colors, spacing, typography... AntD dùng ConfigProvider + theme để override. Nền tảng mọi Design System.

Q: Virtual DOM là gì?

Bản sao nhẹ của Real DOM. React diff Virtual DOM mới vs cũ, chỉ update phần thay đổi (reconciliation). Nhanh hơn manipulate DOM trực tiếp.

Q: useMemo vs useCallback?

useMemo = cache kết quả. useCallback = cache hàm. Tránh re-compute khi re-render. Dùng khi truyền props cho React.memo.

Q: Cross-browser compatibility?

Dùng autoprefixer, check caniuse.com. CJ Desktop → ưu tiên Chrome + Edge (90%+ enterprise users).
